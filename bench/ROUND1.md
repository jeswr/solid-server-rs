## What changed this round

- **Opt #1 — container-listing render O(N²) → O(N)** (`render_container` in `src/ldp/handler.rs`).
  The old `push_unique` did a linear `Vec::contains` scan per child, so building the N `ldp:contains`
  triples was quadratic. Replaced with a `HashSet<Triple>` membership check (seeded from the stored
  body), making each generated-triple push O(1) and the whole render O(N). The generated triples are
  unique by construction (3 distinct typing triples; one containment triple per child, and a child
  appears at most once in the index — both store impls enforce it, and an RDF graph holds a
  containment triple at most once), so the de-dup only ever suppresses a stored↔generated overlap —
  the SAME case the old scan caught. **Insertion order, the triples emitted, the serialised bytes, and
  the representation ETag are unchanged** (a pinning test asserts the multi-child + stored-overlap
  output).
- **Opt #2 — single-pass ACL on read** (`serve_read` + `src/authz/wac.rs`). A GET previously resolved
  the effective ACL once to authorize and then again to build `WAC-Allow` (a second `WacAuthorizer`, a
  second `protected_resource`, and — for an AUTHENTICATED requester — a second full ACL
  walk/read/parse for the public set). The new `WacAuthorizer::authorize_read` resolves the effective
  ACL **exactly once** (`resolve_effective_acl` — the only child→root walk + `store.read` + `oxttl`
  parse) and derives BOTH the access decision (Allow / 401 / 403) AND the `user`/`public` audiences
  from that ONE shared, already-parsed resolution via pure rule-matching (`modes_for`). An
  authenticated read therefore reads + parses the `.acl` ONCE, not twice (the second audience is a
  pure rule-match over the same triples, no I/O). The decision and the header are computed by the SAME
  helpers against the SAME protected resource + the SAME parsed triples, so the gate and the
  advertisement are byte-identical (same 401-vs-403, same fail-closed, same origin-scoped public set);
  an `authorize_read` ↔ old-two-call equivalence test pins this. Conformance stays 41/41.

### (b) GET a container listing — 500 children (the O(N²)→O(N) effect, where it dominates)

| concurrency | BEFORE RPS | AFTER RPS | ΔRPS | BEFORE p50 | AFTER p50 | BEFORE p99 | AFTER p99 |
|---:|---:|---:|---:|---:|---:|---:|---:|
| 16 | 2,181 | 7,857 | **+260% (≈3.6×)** | 6.690 | 1.897 | 20.893 | 5.416 |
| 32 | 2,231 | 7,417 | **+232% (≈3.3×)** | 13.880 | 4.021 | 31.514 | 12.285 |

At 500 children the quadratic term dominated the old render (≈2.2k RPS, p50 ~7–14 ms); the O(N) render
sustains **≈7.4–7.9k RPS at p50 ~2–4 ms** — a **~3.3–3.6× throughput** rise and a **~71% p50 drop**.
The win grows superlinearly with child count, exactly the O(N²)→O(N) signature.

At 10 children BEFORE and AFTER are within run-to-run noise (~3%): with only ~10 members the old

Flat within ±2% (run-to-run noise). This is expected: the public-doc bench is ANONYMOUS, so Opt #2's
public-set derivation is already just `user.clone()` on the anonymous path — there is no second ACL
read/parse to eliminate for an anonymous requester. Opt #2's measurable saving (one full ACL
walk + `store.read` + parse, now collapsed to a single resolution + a pure second rule-match) lands on
the **authenticated** read path, which this auth-free harness cannot drive (the deferred scenario (c)
— needs a Keycloak DPoP token). Opt #2 is measurably neutral on the public-doc ceiling here and
removes a redundant ACL read/parse per authenticated GET; it does not regress the anonymous path.

## $ / cost note

Round 1 is pure compute efficiency on the in-process render + authz path — no S3/QLever request-count
or storage change — so the cost effect is **lower compute per listing GET** (the listing render path
used ~4.9 cores at saturation per the baseline; the O(N) render does materially less work per request
as N grows, which on a right-sized container translates to more listing throughput per vCPU). No
absolute $ figure is asserted (it depends on the deployed container size + traffic mix); the
throughput-per-core gain above is the estimable lever. No new caching across requests was introduced
(that is a later round).
