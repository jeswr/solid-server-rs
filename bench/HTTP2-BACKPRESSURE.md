## (1) HTTP/2 via ALPN — the multiplexing effect

The server now advertises ALPN `[h2, http/1.1]` (owned + tested in `src/tls.rs`; see the commit). An
`h2`-capable client (here `oha --http2`) negotiates HTTP/2; an HTTP/1.1-only client negotiates down.
Verified end-to-end: `curl --http2 …` reports `http_version=2`, `openssl s_client -alpn h2` reports
`ALPN protocol: h2`, and `curl --http1.1 …` still returns `200` over `http/1.1`.

**AFTER (this branch), HTTP/1.1 vs HTTP/2, RPS (success_rate 1.0000 throughout):**

| scenario    | c   | h1 RPS  | h2 RPS  | h2/h1 | h1 p50 ms | h2 p50 ms |
|-------------|-----|---------|---------|-------|-----------|-----------|
| public-doc  | 1   | 1795.8  | 5566.4  | 3.10× | 0.286     | 0.123     |
| public-doc  | 16  | 17374.4 | 14130.8 | 0.81× | 0.822     | 0.838     |
| public-doc  | 64  | 18148.3 | 14766.5 | 0.81× | 3.300     | 3.462     |
| public-doc  | 256 | 17002.0 | 14256.8 | 0.84× | 13.718    | 15.107    |
| listing     | 1   | 1117.4  | 3154.4  | 2.82× | 0.589     | 0.258     |
| listing     | 16  | 9159.5  | 10885.2 | 1.19× | 1.229     | 1.165     |
| listing     | 64  | 10849.7 | 10206.1 | 0.94× | 5.259     | 5.394     |
| listing     | 256 | 10183.6 | 9165.8  | 0.90× | 22.576    | 23.763    |

**Reading it honestly:**
- **The h2 win is at LOW concurrency (one reused connection): ~3.1× (public-doc) / ~2.8× (listing)
  at `c=1`, with ~½ the p50 latency.** This is exactly the case h2 exists for — many small requests
  multiplexed/pipelined over ONE connection with no per-request handshake, vs HTTP/1.1's serial
  request-then-response on a single connection. It is the realistic *many-small-authenticated-
  requests-per-client* shape (a Solid app polling/reading a pod over one kept-alive connection).
- **At high concurrency (`c≥16`) on this CPU-bound, in-memory, loopback workload, h1 is level-or-
  slightly-ahead.** With many parallel connections, h1 already keeps all cores busy; h2's single-
  connection stream multiplexing + HPACK adds a little per-request CPU that does not pay off when the
  bottleneck is already CPU, not connections/handshakes. This is the well-known h2 trade-off — its win
  is connection/handshake amortisation, which the `c=1` rows isolate; it is not a raw-CPU speedup.
- **Net:** h2 is **additive and free to offer** — it is a large win for the connection-amortised
  client shape and never breaks an h1 client (ALPN negotiates down). On a real deployment with TLS to
  remote clients (handshake RTT ≫ loopback) the `c=1` amortisation win is *larger* than measured here.

## (2) Overload / backpressure — does the layer cost throughput? (no)

The new admission-control + request-timeout layers (`src/overload.rs`) wrap the router. Admission is a
single `Semaphore::try_acquire_owned` (one atomic) on the hot path; the default ceiling
(`SOLID_SERVER_MAX_CONCURRENCY=10000`) is far above any level the sweep reaches, so it never trips.

**BEFORE (`main` 79cc7a1, no overload layer) vs AFTER (this branch, layer present), HTTP/1.1 RPS:**

| scenario    | c   | BEFORE RPS | AFTER RPS |
|-------------|-----|------------|-----------|
| public-doc  | 16  | 16372.6    | 17374.4   |
| public-doc  | 64  | 17207.8    | 18148.3   |
| public-doc  | 256 | 15232.8    | 17002.0   |
| listing     | 16  | 8698.5     | 9159.5    |
| listing     | 64  | 9625.7     | 10849.7   |
| listing     | 256 | 10474.0    | 10183.6   |

At the meaningful steady-state concurrencies (`c≥16`) AFTER is within run-to-run noise of (and here
mostly above) BEFORE — **the admission/timeout layer adds no measurable throughput cost** (a
`try_acquire` is one atomic). The `c=1` single-connection rows swing run-to-run (warm/cold TLS) and
are not a reliable capacity signal; the steady-state rows are the no-regression evidence.

### Backpressure behaviour (deterministic, from the tests — not wall-clock)

The shed path is proven deterministically in `tests/overload_http.rs` (and unit-tested in
`src/overload.rs`), not by a flaky load run:
- **At capacity a request is shed with `503` + a jittered `Retry-After`** (base 1s + `0..=4s` jitter,
  to break a retry stampede) and **never reaches the inner stack** — `sheds_with_503_and_retry_after_
  at_capacity`. After the in-flight request releases its permit, the server **recovers** (the next
  request is admitted, not shed).
- **A shed request can NEVER bypass auth/WAC** — `shed_does_not_bypass_auth_for_a_protected_write`
  fires an anonymous PUT at capacity and asserts `503` (shed), never a `2xx`. Admission is the
  OUTERMOST layer, so it sheds *before* auth/DPoP-crypto: a 503 grants strictly less than the request
  would otherwise get (fail-safe by construction).
- **Health probes are shed-EXEMPT** — `health_probes_are_exempt_from_shedding` exhausts the pool and
  asserts `/livez` + `/readyz` still answer `200` (mounted OUTSIDE the layer), so a load balancer's
  readiness probe is never shed (shedding a healthy node's probe would amplify an overload into an
  outage).
- **Graceful drain on SIGTERM** (not just Ctrl-C/SIGINT) — verified live: `kill -TERM` logs
  `SHUTDOWN: SIGTERM received — draining (clean LB deregistration)` and the in-flight drain window
  applies, so an instance behind a load balancer deregisters cleanly.

## Configuration knobs (added this tranche)

| env | default | effect |
|---|---|---|
| `SOLID_SERVER_MAX_CONCURRENCY` | `10000` | global in-flight ceiling; excess ⇒ `503` + `Retry-After`. `0`/invalid ⇒ default (never shed-everything on a typo). |
| `SOLID_SERVER_REQUEST_TIMEOUT_SECS` | `30` | per-request timeout ⇒ `504`. `0` ⇒ disabled. |
| `BENCH_HTTP2` (bench only) | `0` | `1` ⇒ `oha --http2` (drive HTTP/2). |

## Follow-ups (not in this tranche)

- **Authenticated h1-vs-h2 sweep.** `bench/run-auth.sh`'s reqwest client auto-negotiates h2 via ALPN
  now (the server offers it), so the authed path likely already benefits from the same connection
  amortisation; an explicit `AUTH_HTTP2` toggle (`reqwest …http2_prior_knowledge()`) for a clean
  side-by-side authed h1-vs-h2 number is a small, mechanical follow-up.
- **`/metrics` exporter.** `AdmissionMetrics` (in-flight gauge + shed counter) are plain atomics ready
  for a Prometheus/OpenTelemetry exporter; only the log-on-shed is wired today.
- **`/readyz` real backend check.** Today `/readyz` is a static "process up"; when the live
  SPARQ/object_store store lands it should 503 on backend-unreachable so a not-ready instance
  deregisters cleanly behind the LB (the seam is in `app::health_routes`).
