## Saturation + concurrency findings

- **Public-doc throughput saturates at ~c=8–16 → ≈44–45k RPS** (best-of-3 at c=16: 44.9k / 41.4k /
  39.8k). Beyond ~c=16 throughput is FLAT-to-slightly-declining while latency climbs linearly with
  concurrency (p50 0.08 ms → 12 ms from c=1 → c=512) — the classic past-the-knee behaviour. No errors
  at any level up to c=512.
- **Listing throughput saturates at ~c=16 → ≈20–21k RPS** (best-of-3 at c=16: 21.7k / 20.4k / 20.2k)
  — roughly **2.2× lower than the public doc**. The container-membership render is the per-request
  cost difference (both share the identical TLS + WAC + store-read path).
- **The server is NOT CPU-bound on the public-doc path** — during a saturated public-doc c=32 run the
  server process used **≈336% CPU (~3.4 of 8 cores)** while `oha` consumed the rest of the box. On the
  LISTING c=32 run the server used **≈488% CPU (~4.9 cores)** — markedly MORE per-request compute,
  consistent with the listing being a server-side render cost, not a transport cost. On this
  single-box loopback setup, the public-doc plateau is bounded by client/loopback/TLS round-trip
  contention (there is server headroom); the listing plateau is bounded by server render compute.
- **The listing render cost scales with child count.** Same harness, `SOLID_SERVER_SEED_BENCH=10`
  (10 children) vs `=100`, both c=16: **10 children → ≈37.7k RPS (p50 0.36 ms)** vs **100 children →
  ≈20.7k RPS (p50 0.71 ms)**. A 10× larger membership ≈ halves throughput — the listing path's cost
  is dominated by the per-child render work, not a fixed overhead.

## Hot-path analysis — where the time goes per op

The GET hot path (`src/ldp/handler.rs::serve_read`) for an anonymous public read is:

1. **TLS** (amortised across keep-alive — `oha` reuses connections, so the handshake is paid once per
   connection, not per request; the public-doc p50 of 0.08 ms at c=1 shows the warm-connection
   per-request cost is tiny).
2. **Auth middleware** (`src/auth.rs`) — for an anonymous request the verifier returns a *public*
   token with **no JWKS fetch and no crypto** (an absent `Authorization` header short-circuits before
   any signature/DPoP work). So public reads pay ~nothing here. (The authed path — scenario c — is
   where token-verify cost lives; unmeasured, see above.)
3. **WAC authorization** (`src/authz/wac.rs`) — runs on EVERY read. It reads the target's own `.acl`
   through the store (a `NotFound` for most resources), then walks ancestors reading each `.acl` until
   one is found, then parses it (`oxttl`) into triples and computes modes. For the bench public doc
   this is: read `…/doc.acl` (miss) → read `…/public/.acl` (miss) → read `…/bench/.acl` (HIT) → parse
   → `modes_for`. It then resolves the public set AGAIN for `WAC-Allow` (`effective_permissions`),
   re-walking + re-reading + re-parsing the SAME ACL a second time.
4. **Store read** (`src/store/mod.rs::read`) — a `Mutex<HashMap>` metadata lookup + a `Mutex<HashMap>`
   blob fetch (two lock acquisitions, both cheap and uncontended at these rates).
5. **Container render** (listing only — `render_container`) — parse the container's stored body, then
   for each of N children build a `Triple` and `push_unique` it (a **linear `Vec::contains` scan per
   child → O(N²)**), then `serialize_triples` (oxttl) the whole set.
6. **Response assembly** — ETag (FNV-1a over the body for a container), `WAC-Allow`, discovery/type/acl
   `Link` headers, Range handling.

### Top optimization targets (ranked, with a hypothesis each)

1. **Container-listing render is O(N²) in the child count + re-parses the stored body
   (`render_container` / `push_unique` in `src/ldp/handler.rs`).** RANK 1 — this is the largest,
   most-scalable win and the measured 2.2× gap + the 10-vs-100-children halving point straight at it.
   `push_unique` does `triples.contains(&triple)` (a linear scan) for every child, so building the
   `ldp:contains` set is quadratic; the listing also re-parses the container's stored body each GET
   and re-serialises the full triple set. *Hypothesis:* replace the linear-scan dedup with a
   `HashSet`-backed membership check (or skip dedup for the generated containment triples, which are
   unique by construction — only the small stored-body overlap needs a check), making the render O(N).
   Expected: the listing throughput curve flattens toward the public-doc curve as N grows; the
   10→100-children halving largely disappears.

2. **WAC reads + parses the effective ACL TWICE per read (once to authorize, once for `WAC-Allow`).**
   RANK 2 — applies to EVERY GET/HEAD (public and listing), so it lifts the whole read floor.
   `serve_read` calls `authorize(...)` (walks + reads + parses the ACL) and then
   `wac_allow_value` → `effective_permissions(...)` which, for an anonymous requester, re-resolves the
   SAME protected resource (a second ancestor walk, a second `store.read` of the same `.acl`, a second
   `oxttl` parse). *Hypothesis:* thread the already-resolved effective-ACL triples (or at least the
   resolved ACL IRI + parsed triples) from the authorize step into the `WAC-Allow` computation so the
   ACL is read + parsed ONCE per request. Expected: a measurable per-request latency drop on BOTH
   scenarios, larger as ACL bodies grow; no behaviour change (same modes, same header).

3. **The ACL ancestor walk issues a separate `store.read` per ancestor `.acl` (sequential awaits) and
   re-parses on every hit.** RANK 3 — complements #2. Each read is a `Mutex` lock + clone; for a deep
   path that is several sequential lookups, and the parse of the found ACL is repeated on every
   request to the same resource. *Hypothesis:* an ETag-keyed parsed-ACL cache (the cost lever the
   charter calls out — cache ACL decisions keyed by the ACL resource's ETag) would turn the
   read+parse into a hash lookup for the hot (repeated-target) case, and is the seam toward the SPARQ
   "ask for the per-resource decision" model. Expected: large win for repeated reads of the same
   resource/subtree; needs an invalidation story (ETag-keyed) so it never serves a stale decision —
   correctness-first, measure before/after.

4. **Per-response header/string allocation on every read (`format!` ETag, `WAC-Allow` string,
   discovery `Link` headers).** RANK 4 — small, broad. Each read formats several strings + a fresh
   `HeaderMap`. *Hypothesis:* precompute/cache the static discovery + type `Link` header values (they
   don't vary per request) and avoid re-formatting invariant headers. Expected: a modest constant-factor
   improvement on the public-doc (ceiling) path where per-request fixed cost dominates; lowest priority
   until #1–#3 land.
