# solid-server-rs — AUTHENTICATED (DPoP) HTTPS load benchmark BASELINE

> **Generated by the committed harness `bench/run-auth.sh`** (server boot + Keycloak DPoP-token mint
> + a Rust per-request-DPoP-proof load client + the concurrency sweep). The numbers below are REAL
> measurements — not estimates — captured by that script on the machine + date stated. Re-run with
> `./bench/run-auth.sh`; the authoritative raw data is `bench/results-auth/{authed,anon}.tsv` + the
> per-level `bench/results-auth/*.json` (gitignored — regenerate). This is the round-2 **auth-path**
> baseline to be minimised in round 3; it changes **no server request-handling behaviour** (the only
> server `src/` touch is a default-off, conformance-neutral DPoP-replay-capacity env knob — see below).

This is the round-2 companion to the **anonymous** baseline in [`bench/BASELINE.md`](./BASELINE.md).
The authenticated GET is the realistic **production** path: the server pays DPoP-proof verify +
access-token verify + a JWKS-cache lookup + token parse + the single effective-ACL read/parse (round-1
Opt #2 lands its saving HERE) + the store byte fetch + response assembly. The auth-free `oha` harness
in rounds 0–1 could not drive it (it cannot mint a fresh RFC 9449 DPoP proof per request), so this
harness closes that gap.

### How the token + DPoP proof are built (reused from conformance — no new auth flow)

- **Token (once):** the realm requires a DPoP proof ON THE TOKEN REQUEST (DPoP-bound client
  credentials). The client mints a token-endpoint proof (`htu`=token endpoint, `htm`=POST, no `ath`),
  POSTs `grant_type=client_credentials` (`conformance-alice` / `…-secret`), and receives a
  `cnf.jkt`-bound `at+jwt` whose `webid` is `https://localhost:3000/alice/profile/card#me` and `aud`
  includes the server base URL (decoded live: `aud=["solid","https://localhost:3000"]`).
- **Proof (per request):** a fresh proof from the SAME ES256 key — `htu`=exact request URL,
  `htm`=GET, unique `jti`, `iat`, and **`ath`=base64url(SHA-256(access_token))** (the server enforces
  `ath`). The `jti` carries a per-process random nonce so a fresh client run against an already-running
  server never collides with a prior run's jtis in the replay store.
- **Pre-flight (before every sweep):** a single authed GET MUST be **200** AND an anonymous GET MUST
  be **401** (genuinely owner-private), else the run aborts — so the sweep can never silently measure
  the 401 path or a public read instead of the auth path.

## Auth overhead vs anonymous (the delta = DPoP verify + token verify + JWKS lookup + the ACL read/parse)

Measured on the SAME box/binary/run (auth client WITH token vs the same client WITHOUT):

| metric | anonymous | authenticated (c) | overhead |
|---|---:|---:|---:|
| **per-request latency, c=1, p50** | 0.083 ms | 0.167 ms | **+0.084 ms (≈ 2.0×)** — the marginal cost of one authed request on an idle server |
| **per-request latency, c=1, p99** | 0.153 ms | 0.340 ms | +0.187 ms |
| **max sustained RPS (saturation)** | ~36.8k (c=16–32) | **~11.7k (c=16)** | **≈ 3.1× lower throughput** |
| **saturation concurrency** | c≈16–32 | c≈16 | similar knee |

- **The authed path saturates at the same concurrency (~c=16) but at ≈1/3 the throughput.** The extra
  per-op compute (the two ES256 signature verifies + the JWKS-cache lock/clone + the ACL read/parse +
  proof JSON parse) is the ~0.08 ms marginal cost that, fully pipelined, caps RPS ~3× below the
  anonymous read.
- **The private-LISTING path (d) is ~22% below the private-DOC path (c)** at the c=16 knee (9.1k vs
  11.7k RPS) — the container-membership RDF render is the per-request difference, on TOP of the auth
  cost, exactly mirroring the anonymous listing-vs-doc gap in `bench/BASELINE.md`. Round-1's O(N)
  listing fix already applies here; the residual gap is the render + serialise of 100 children.
- **CPU:** the box was kept quiet for measurement; at the authed saturation point the server +
  load-client together used the whole box (8 cores), with the authed runs visibly more
  server-CPU-bound per request than the anonymous run (the crypto verifies are the added work). The
  single-box loopback topology means the absolute RPS ceiling is partly the client+TLS+loopback
  round-trip, not purely server compute — the auth *overhead delta* (same client both runs) is the
  trustworthy figure.

## Replay-store saturation — a hard single-instance auth ceiling (round-3-relevant finding)

The in-memory DPoP-replay store (`InMemoryReplayStore`) is capped at **100,000** live `jti`s within
the proof-age TTL window (`DPOP_PROOF_MAX_AGE_SECS` 300 + clock tolerance 5 = **305 s**) and **fails
closed** when full — it never evicts a live `jti` (a correct security posture; evicting would re-open
the replay window). Under SUSTAINED authenticated load this is a hard ceiling, MEASURED with the
production default cap (no override):

> At **c=16, ~10.7k authed RPS**, the 100k store filled in **~9.4 s**: the first 10s window's success
> dropped to **0.937**, and the next two back-to-back windows were **0.000 success** (every further
> proof rejected, fail-closed, for the remainder of the 305s TTL window). A fresh single authed GET
> after saturation also fails — confirming the store stays full for the window. The rejection is the
> verifier's replay-store-unavailable status (a 503), not a 401.

The steady-state, sustainable authed rate the 100k/305s store supports indefinitely is **~328 proofs/s
per instance** (100_000 ÷ 305 s). Above that, a single instance eventually saturates. The baseline
tables above were therefore measured with the dev/bench override `SOLID_SERVER_REPLAY_MAX_ENTRIES`
raised so the sweep measures the genuine verify path rather than the fail-closed path. **This is a real
production behaviour, not a harness artifact** — and it ranks as a round-3 scaling target (below).

> **The override is conformance-neutral + default-off.** `SOLID_SERVER_REPLAY_MAX_ENTRIES` UNSET ⇒ the
> 100_000 production default (`InMemoryReplayStore::with_window`), byte-identical to before. It ONLY
> raises a capacity number — no request-handling logic changes, the fail-closed semantics are
> untouched, and `0`/invalid keeps the default (so a typo can never weaken replay protection). It must
> NEVER be set in production; the shared (Redis `SET NX`) `ReplayStore` is the real horizontal-scaling
> seam.

## Hot-path analysis — where the time goes per AUTHENTICATED op

The authed GET hot path, from the verifier's documented verify order (`solid-oidc-verifier`
`verifier.rs`) + this server's `src/auth.rs` adapter + `src/authz/wac.rs`:

1. **Auth middleware (`src/auth.rs`)** reconstructs the request `htu` (`parse_target`) and calls
   `Verifier::verify`, which runs, in order:
   - scheme parse (DPoP vs Bearer) + the DPoP-required policy check;
   - **access-token signature verify** — an **ES256 (asymmetric) verify** of the `at+jwt` against the
     issuer's JWKS, plus RFC 9068 claim checks (`typ=at+jwt`, `iss`, `aud`, temporal, `cnf`);
   - **DPoP-proof signature verify** — a **SECOND ES256 verify** of the proof against its embedded
     public JWK, plus `htm`/`htu`/`iat`-freshness/`jti`/`ath` checks and the `cnf.jkt`==jkt(proof-JWK)
     thumbprint match (RFC 7638 SHA-256);
   - **jti replay mark** — a `Mutex<HashMap>` check-and-set (fail-closed at capacity — see above);
   - bidirectional WebID↔issuer — **skipped** here (`BIDIRECTIONAL=off`); in `strict` it adds a WebID
     profile fetch (a NETWORK round-trip — the dominant cost when enabled).
   - The **JWKS** is fetched once per issuer and CACHED (`NetworkJwksProvider`: `Mutex<HashMap<issuer,
     (Vec<Jwk>, fetched_at)>>`, 300s TTL) — so the steady state is NOT a network fetch, but EVERY
     request takes the cache `Mutex` lock and **clones the `Vec<Jwk>`** keyset.
2. **WAC authorization (`src/authz/wac.rs`)** — round-1 Opt #2 already made the read path resolve the
   effective ACL ONCE (authorize + `WAC-Allow` share the parse). For `/alice/test/bench-private` it
   walks `…/bench-private.acl` (miss) → `…/test/.acl` (miss) → `…/.acl` (HIT, the owner `acl:default`)
   → parse (`oxttl`) → `modes_for`. The ancestor walk is N sequential `store.read`s; the found ACL is
   re-parsed on every request to the same resource.
3. **Store read** — `Mutex<HashMap>` metadata + blob fetch (cheap, uncontended at these rates).
4. **Container render** (scenario d only) — the round-1 O(N) `ldp:contains` build + `oxttl` serialise.
5. **Response assembly** — ETag, `WAC-Allow`, discovery/type/acl `Link` headers, Range.

### Top auth-path optimization targets for ROUND 3 (ranked, with a hypothesis each)

1. **Two ES256 signature verifies per request (access token + DPoP proof) — the dominant marginal
   auth cost.** RANK 1. Every authed request does asymmetric crypto twice (the ~0.08 ms c=1 overhead +
   the ~3× RPS gap point straight at it). *Hypotheses:* (a) **cache the verified-access-token decision
   keyed by the token string** (the token is constant across a session's many requests; its signature
   + RFC 9068 claims only need verifying once until `exp`) so a repeat presentation skips the
   token verify — only the per-request DPoP proof (which MUST be fresh) is verified each time; this
   roughly **halves the crypto** on the common "many requests, one token" pattern. (b) confirm the
   aws-lc-rs P-256 verify path is the one in use (it is the fastest backend). Expected: a large
   per-request latency drop + RPS lift on repeated authed reads; correctness-preserving (a token
   whose `exp` passed or whose signature was never valid still fails — cache only the *positive*
   verified result, ETag/`exp`-bounded, fail-closed on miss). Measure on scenario (c).

2. **JWKS-cache `Mutex` lock + `Vec<Jwk>` clone on EVERY request.** RANK 2 — applies to every authed
   request. The keyset is cached (no fetch) but each verify locks the cache `Mutex` and clones the
   `Vec<Jwk>`. *Hypothesis:* hold the keyset behind an `ArcSwap`/`RwLock` and hand out an `Arc<[Jwk]>`
   (clone = a refcount bump, no Vec copy), or pass the keys by reference into the verify. Expected: a
   small constant-factor per-request win that grows with keyset size + concurrency (lock contention at
   c≥64). Upstream-shaped (the change is in `solid-oidc-verifier`'s `NetworkJwksProvider`) — file it
   there.

3. **ETag-keyed parsed-ACL cache (the charter's named cost lever).** RANK 3 — complements round-1's
   single-pass: the effective ACL is read ONCE per request but still **re-read from the store +
   re-parsed (`oxttl`) on every request to the same resource/subtree**. *Hypothesis:* cache the parsed
   ACL (the resolved ACL IRI's parsed triples / computed modes) keyed by that ACL resource's ETag, so
   the hot repeated-target case becomes a hash lookup; invalidate on the ACL's ETag changing (never
   serve a stale decision — correctness-first). Expected: a large win for repeated authed reads of the
   same resource/subtree (the common app pattern); the seam toward the SPARQ "ask for the per-resource
   decision" model. Measure on (c) repeated-target.

4. **Replay-store contention + the 100k fail-closed ceiling (scaling, not single-op latency).** RANK 4
   — the `Mutex<HashMap>` jti mark is a single lock per request (contends at high concurrency), AND
   the 100k/305s cap is a hard single-instance ceiling (measured above: ~328 sustained proofs/s before
   fail-closed). *Hypothesis:* this is the **horizontal-scaling seam**, not a micro-opt — a shared
   (Redis `SET NX`) `ReplayStore` removes the per-instance cap and the in-process lock; for a single
   instance, a sharded/lock-striped in-memory store cuts the mark contention. Expected: removes the
   sustained-throughput ceiling (the most important production scaling fix) + reduces tail latency at
   c≥128. Correctness-critical — keep the atomic check-and-set + fail-closed semantics.

5. **DPoP-proof + token JSON/base64 parsing (peek + decode, done more than once).** RANK 5 — small,
   broad. The verifier peeks the header/claims/issuer before the full verify (a base64 + serde_json
   parse), then re-decodes during verification. *Hypothesis:* parse each JWT segment once and thread
   the decoded parts through. Expected: a modest constant-factor improvement; lowest priority until
   #1–#4 land. Upstream-shaped (in the verifier).
